1. EXECUTIVE SUMMARY & VERDICT
================================================================================
HalluciGuard is a rigorous, highly insightful investigation into LLM hallucination
detection utilizing the HaluEval QA benchmark dataset. Rather than simply training
a model and reporting high metrics, this project acts as a critique of the underlying
data benchmark. 

The audit reveals a project with an exceptionally strong scientific foundation:
it uncovers a massive dataset confound (length bias) where hallucinated answers are,
on average, nearly five times longer than correct answers. Consequently, a simple
2-parameter "Length-Only" model (94.35% accuracy) out-performs standard content-based
models (91.90% accuracy). The project also correctly identifies and remediates
train-test split leakage occurring when data expansion is done prior to splitting.

However, the project is severely undermined by repository structure issues:
- There are 44 files at the root level, making it look cluttered and disorganized.
- 27 files remain uncommitted, leading to a massive gap between the local workspace
  and the remote GitHub state.
- Multiple duplicate text cleaning functions and redundant plotting scripts exist.
- Standard folders like src/, data/, docs/, and models/ are missing or empty.

VERDICT: This is a highly portfolio-worthy and Amazon ML Summer School-worthy project
in terms of intellectual and scientific content, but it requires immediate, focused
refactoring and repository cleanup to look professional to recruiters and maintainers.

2.2 Methodological Rigor & Confound Analysis
--------------------------------------------
1. Data Leakage Remediation:
   In HaluEval, each source record contains a Knowledge context (K), a Question (Q),
   a Right Answer (A_right), and a Hallucinated Answer (A_halluc). If we expand this
   directly into binary pairs:
     - Pair 1: (K, Q, A_right) -> label=0
     - Pair 2: (K, Q, A_halluc) -> label=1
   and then perform a random split at the binary sample level, Pair 1 might end up
   in the training set and Pair 2 in the test set. Because they share the identical
   (K, Q) context, the model will memorize context details rather than learning
   generalizable factual verification.
   The project successfully fixes this by splitting the 10,000 paired rows first
   (8,000 train, 2,000 test) and then expanding them into binary sets (16,000 train,
   4,000 test). This drops overlap leakage from 32.24% down to exactly 0.00%.

2. Exposing the Length Confound:
   The length-based exploratory analysis reveals that:
     - Factual answers are extremely short (mean: 13.65 characters, 2.22 words).
     - Hallucinated answers are much longer (mean: 66.22 characters, 10.98 words).
   A simple Logistic Regression trained on only two features—character count and word
   count—achieves 94.35% accuracy and an F1-score of 0.9430 on the test set. 
   This length model outperforms the text content model (TF-IDF on Answer-Only) which
   scores 91.90% accuracy. This indicates that the HaluEval QA benchmark is heavily
   biased; the models are not learning factuality or semantic coherence, but are
   primarily acting as sentence-length classifiers.

3. Input Format Ablation:
   The ablation study yields a fascinating and counter-intuitive result:
     - Answer-Only: 91.90% Accuracy (F1: 0.9161)
     - Question + Answer: 74.10% Accuracy (F1: 0.7319)
     - Full (Knowledge + Question + Answer): 64.30% Accuracy (F1: 0.6165)
   Adding more context (Question and Knowledge) actually hurts the TF-IDF model.
   This is because the Knowledge context is very long and identical across factual
   and hallucinated versions of a question. It acts as massive textual noise,
   confounding the Logistic Regression classifier which tries to associate terms in
   the Knowledge context with the labels, leading to overfitting.

4. Cross-Validation Vocabulary Leakage:
   The cross_validation_study.py script performs a stratified 5-fold CV. However,
   it fits the TfidfVectorizer on the entire training set before executing the CV
   folds loop. This is a classic methodology slip: vocabulary definition and IDF
   frequency calculations leak from validation folds into training folds. Though the
   practical impact is minimal due to the large dataset size, this is a research gap
   that must be noted.

2.3 Clarifying Answers to Core Audit Questions
----------------------------------------------
* Is this a legitimate ML project?
  Yes. It contains a complete pipeline, handles a real-world benchmark, and provides
  a deep analysis of dataset artifacts.

* Is it resume-worthy?
  Highly. It demonstrates the ability to critique datasets and catch methodological
  errors. This is a critical skill for Principal ML Engineers and Research Scientists
  that distinguishes them from beginners.

* Is it Amazon ML Summer School worthy?
  Yes. It perfectly aligns with the curriculum's emphasis on data rigor, leakages,
  and baselines.

* Is the ML portion complete?
  Yes, the pipeline is complete. However, from a modeling standpoint, it lacks a
  deep learning baseline (e.g. DistilBERT or RoBERTa), which is standard in 2026.

* Are additional models actually necessary?
  Yes, to determine whether deep learning models are also tricked by the length confound,
  or if they can successfully learn the semantics of factuality independent of length.

* What are the strongest and weakest technical aspects?
  - Strongest: Leakage detection/remediation, baseline comparison with length features,
    comprehensive ablation study, and qualitative error analysis.
  - Weakest: Vocabulary leakage in CV, hardcoded code redundancy, and poor file hygiene.

* What scientific contribution or insight does the project provide?
  It proves that the HaluEval QA benchmark is highly flawed due to generating
  hallucinated answers that are systematically longer than correct answers. Any model
  trained on it must control for sentence length to claim true semantic factuality.

-----------------------------------------
- Flat Directory Structure: 40+ scripts and configuration files sit at the root directory
  level. This is a severe repository layout violation for any professional project.
- Massive Gap in Commit History: The current GitHub repository has 17 commits, but they
  were all made in a 6-minute window on June 11, 2026. Since then, 27 files (including
  key result scripts, plots, metrics, and models) remain uncommitted.

  `extract_parts()` are duplicated verbatim across 14+ different scripts. This is extremely
  poor software engineering practice.
- Plotting Inefficiencies: There are 12 different python files each plotting a single metric,
  rather than a unified plotting script or a visualization module.
- Missing Standard Files: The repository lacks a LICENSE file.
